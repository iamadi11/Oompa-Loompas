# Decision: Release 0.1.6 (`@oompa/web`)
Date: 2026-04-06
Phase: Phase 1 — Deal + Payment Intelligence (client trust + shell quality)
Status: APPROVED

## What
Cut **`@oompa/web@0.1.6`**: accessibility and documentation updates for the **home overview unavailable** experience, plus **release-facing docs** (changelog, release notes, product research backlog) so CI/CD and humans share one definition of “what shipped.”

## Why now
**0.1.5** tagged the state-machine split (unavailable vs empty); **0.1.6** carries the follow-up **a11y landmark** work and **browser MCP checklist** used as a release gate. Without a version bump and consolidated notes, deploy artifacts and support cannot reference a single label.

## Why not merge into 0.1.5 retroactively
The registry version **0.1.5** was already associated with the first slice; republishing the same semver with different bits breaks reproducibility. Forward-only **0.1.6**.

## Success criteria
- `apps/web/package.json` version **0.1.6**
- [CHANGELOG.md](../../CHANGELOG.md) section **[0.1.6]**
- [docs/releases/RELEASE-0.1.6.md](../releases/RELEASE-0.1.6.md) complete
- Research backlog filed under [docs/product/feature-candidates-2026-04.md](../product/feature-candidates-2026-04.md)

## Assumptions
- **`@oompa/api`** remains **0.1.0** (no API contract change in this release).
- Production/staging deploys use existing [infra/README.md](../../infra/README.md) workflows after **CI green**.
