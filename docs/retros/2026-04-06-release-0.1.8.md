# Retro: Release 0.1.8 (CORS dev 3005 + invoice a11y polish)
Shipped: 2026-04-06

## What was built
**API:** Development CORS allowlist extended for **Next.js `dev:clean`** (**3005**, **localhost** and **127.0.0.1**). **Web/utils:** Invoice link and HTML accessibility improvements aligned with existing payment invoice v1 UX doc.

## Decisions made (and why)
- **Explicit allowlist** instead of reflecting arbitrary Origins in dev — keeps behavior deterministic and reviewable under SOT “trust > intelligence.”
- **No production behavior change** — production still uses **`WEB_URL`** only to avoid broadening cross-origin access.

## What the critic user said
Busy creators do not see CORS; they see a broken tab. Fixing dev parity prevents the team from shipping false negatives when validating **get paid** flows.

## Post-deploy baseline
No new production metrics for CORS. Invoice checklist remains the manual browser validation artifact for printable HTML.

## What to watch
- Support questions: “invoice works in prod but not locally” → confirm **`API_URL`** / **`NEXT_PUBLIC_API_URL`** and which dev port is in use.
- If new official dev ports are added, extend **`DEV_WEB_ORIGINS`** and tests in the same commit.

## What we’d do differently
Optional: single integration test matrix for all **`DEV_WEB_ORIGINS`** entries to avoid one-off port additions.
